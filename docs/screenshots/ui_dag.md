# DAG-визуализация плана подзадач

> Изображение экрана прилагается отдельным файлом. Ниже — текстовое описание и анализ.

## Что показано

Вкладка `План` отображает **DAG-визуализацию** (направленный ациклический граф)
подзадач, на которые `Orchestrator` декомпозировал требование. Каждый узел графа —
подзадача с полем `why` (обоснование), а рёбра показывают зависимости порядка
выполнения.

На экране — план для завершённой задачи типа `add_entity`: граф из **4 подзадач**
(что соответствует индикации «4/4» на главном экране `Agents`). Узлы окрашены по
статусу выполнения (выполнено / в работе / ожидает), что даёт наглядную картину
прогресса.

JSON-план от `Orchestrator` имеет вид:

```json
{
  "subtasks": [
    { "id": 1, "title": "Создать класс сущности CustomerCategory", "why": "..." },
    { "id": 2, "title": "Добавить 4 поля и self-referencing FK",      "why": "..." },
    { "id": 3, "title": "Зарегистрировать ассоциацию в Initializer",  "why": "..." },
    { "id": 4, "title": "Добавить EF Core-конфигурацию",              "why": "..." }
  ]
}
```

## Анализ

DAG-представление делает план машины прозрачным для человека: до утверждения через
`ApprovalGate` пользователь видит не только список действий, но и их
взаимозависимости и обоснования. Согласованность графа с метрикой «4/4» главного
экрана подтверждает целостность отображения — интерфейс показывает одно и то же
число подзадач во всех местах. Поле `why` каждого узла — тот же текст, что попадает
в `subtask_why` журнала аудита.
