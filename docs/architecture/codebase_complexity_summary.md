# Анализ сложности кодовой базы .NET-приложения

## Назначение документа

Данный документ демонстрирует масштаб и архитектурную сложность целевого .NET 3.1 приложения, с которым работает мультиагентная система AI Junior Developer. Сложность кодовой базы напрямую определяет требования к точности анализа кода, качеству планирования изменений и корректности генерации кода агентами.

---

## 1. Масштаб проекта

| Метрика | Значение |
|---------|----------|
| Всего `.cs` файлов | **1 559** |
| Классы сущностей (Entities) | **253** |
| Сервисы (Services) | **322** |
| REST-контроллеры (Controllers) | **34** |
| Типы сообщений MassTransit | **143** |
| Enum-типы | **50+** |
| Проекты в решении | **4** (WebApi, Data, Filtering, XsltProcessor) |

Приложение представляет собой корпоративную систему управления строительными проектами с финансовым учётом, аудитом, аналитикой и интеграциями с внешними сервисами.

---

## 2. Архитектурные паттерны

Кодовая база построена на базе корпоративного фреймворка **Visary**, включающего 10+ модулей (Core.DAL, Workflow, Security, Approval, Calendars, Employee, Notification и др.), что создаёт дополнительный уровень абстракций, который агенты должны понимать и соблюдать.

**Применяемые паттерны:**
- **Domain-Driven Design** — 253 доменные сущности с глубокой иерархией наследования (до 4 уровней: `ConstructionSite → BaseSyncProject → BaseSyncObject → BaseObject`)
- **Service Layer** — 322 сервиса, из которых 94 CRUD-сервиса на основе 9 базовых классов-шаблонов
- **Event-Driven Architecture** — обработчики событий MassTransit для каскадных вычислений
- **Strategy Pattern** — подключаемые стратегии для каскадного выбора, хранилища файлов, аудита
- **Template Method** — `AbstractExcelLoaderService<T>` с 8 конкретными реализациями импорта

## 3. Сложность модели данных EF Core

### 3.1. Иерархия сущностей
Каждая сущность наследует от одного из базовых классов и реализует **4–6 интерфейсов** для сквозной функциональности:

```
ConstructionSite : BaseSyncProject,
    IDetailViewByCriteriaObject,
    IAdditionalPropertiesObject,
    IWorkflowObject,
    IConstructionProjectObject,
    IGeoObject
```

### 3.2. Навигационные свойства и связи
Крупнейшие сущности содержат **60+ свойств**, включая:
- Простые типы (`string`, `int`, `decimal`, `DateTime`)
- FK-ссылки на другие сущности (`public AnalogType AnalogType { get; set; }` + `public int? AnalogTypeID { get; set; }`)
- Коллекции (`ICollection<T>`)
- Enum-поля (`ApprovalStatus`, `ImportStatus`, `ProjectStage`)
- Вычисляемые свойства через `CompiledExpression`

### 3.3. Централизованная конфигурация
Вся регистрация ~150+ сущностей и их ассоциаций сосредоточена в **одном файле `DataConfiguration.cs` (907 строк)** через fluent API:

```csharp
.Entity<ConstructionSiteAnalog>(s => s
    .AssociatedEntity(f => f.AnalogType)
    .AssociatedEntity(f => f.FloorCategory))
```

**Критическое ограничение:** при добавлении навигационного свойства агент обязан зарегистрировать ассоциацию в `DataConfiguration.cs` и дополнительные связи в `Initializer.cs`. Пропуск любого из шагов приводит к ошибке сборки или некорректному поведению в runtime.

---

## 4. Почему это сложно для ИИ-агентов

### 4.1. Контекстная зависимость изменений
Добавление одного поля в сущность может потребовать изменений в **5–7 файлах**:
1. Класс сущности (`Data/Entities/`)
2. Конфигурация EF Core (`DataConfiguration.cs`)
3. Инициализатор (`Initializer.cs`)
4. CRUD-сервис (`Data/Services/CrudServices/`)
5. DTO-модель и маппинг
6. Контроллер (при необходимости)
7. Фильтры и сортировки (`Filtering/`)

### 4.2. Неявные правила и конвенции
- FK-свойства обязательно используют `ID` (заглавные), а не `Id`
- Регистрация ассоциаций зависит от типа связи (one-to-many vs. one-to-one) и выполняется на стороне **родительской** сущности
- Enum и примитивные типы **не требуют** регистрации в `DataConfiguration`, а навигационные свойства — **обязательно** требуют
- Запрещено создавать отдельные `{Entity}Configuration.cs` файлы — вся конфигурация централизована
- Запрещено вручную добавлять `DbSet<T>` в `DataContext` — это делает базовый фреймворк Visary

### 4.3. Масштаб поиска при анализе
При планировании изменений агент-оркестратор должен проанализировать:
- **253 класса сущностей** для поиска связанных доменных объектов
- **322 сервиса** для обнаружения затронутых бизнес-операций
- **143 типа сообщений** для учёта событийных обработчиков
- **907-строчный конфигурационный файл** для определения порядка регистрации

Ручной поиск по тексту в таком объёме кода неэффективен, поэтому агенты используют **статический анализатор Roslyn** для точного построения графа зависимостей: определения иерархии классов, навигационных свойств, мест вызова методов и конфигурационных точек.

### 4.4. Хрупкость сборки
Приложение на .NET 3.1 с централизованной конфигурацией EF Core крайне чувствительно к ошибкам:
- Пропущенная регистрация ассоциации → `InvalidOperationException` при запуске
- Дублирование сущности → `CS0101` (ошибка компиляции)
- Неверный тип связи → некорректная миграция БД
- Нарушение иерархии наследования → каскадные ошибки во всех зависимых сервисах

Именно поэтому в системе реализован итеративный цикл сборки и исправления ошибок с контейнеризацией через Docker.

## 5. Импорт/экспорт данных

Отдельный блок сложности — **8 специализированных загрузчиков Excel** и **19 парсеров**, каждый из которых реализует шаблонный метод `AbstractExcelLoaderService<T>` с собственной логикой маппинга колонок, валидации и трансформации данных. Доработка алгоритмов импорта требует понимания как доменной модели, так и структуры входных файлов.

## 6. Интеграционные точки

Приложение интегрировано с:
- **MassTransit** — асинхронная шина сообщений с 143 типами контрактов
- **Hangfire** — фоновые задачи по расчётам
- **PostgreSQL + PostGIS** — геопространственные данные
- **Yandex API** — внешние сервисы
- **Public Alpha** — интеграция с внешними системами
- **XSLT-процессор** — генерация отчётов через XML-трансформации

## 7. Выводы

Кодовая база .NET-приложения характеризуется:

1. **Высоким масштабом** — 1559 файлов, 253 сущности, 322 сервиса
2. **Глубокой иерархией наследования** — до 4 уровней + множественная реализация интерфейсов
3. **Строгими архитектурными конвенциями** — нарушение любой из них приводит к ошибкам сборки
4. **Централизованной конфигурацией** — один файл 907 строк для регистрации всех сущностей
5. **Высокой связанностью компонентов** — изменение одной сущности затрагивает 5–7 файлов
6. **Множеством неявных правил** — не задокументированных в коде, но обязательных для соблюдения

Всё это делает задачу автоматической доработки кода агентами нетривиальной и требует:
- **Roslyn-анализатора** для точной навигации по коду и построения графа зависимостей
- **RAG-системы** с историческими примерами успешных доработок из GitLab
- **Специализированных агентов** (оркестратор, кодер, эксплорер, тестировщик) с чётким разделением ответственности
- **Итеративного цикла тестирования** с автоматической сборкой в Docker-контейнере
- **Шаблонов доработок** для различных типов задач (добавление сущностей, модификация сервисов, импорт/экспорт и др.)
