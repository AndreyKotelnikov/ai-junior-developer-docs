# Обоснование выбора единственного типа задач `modify_entity`

Документ обосновывает решение сосредоточить факторный эксперимент на единственном типе задач `modify_entity` (модификация существующей сущности EF Core с добавлением полей, навигационных свойств и связей), исключив из тестового набора другие 9 типов задач корпоративной кодовой базы.

## Распределение задач корпуса MR-diff по типам

По результатам сплошного анализа 1 154 пар MR-diff корпоративного GitLab за 2023–2025 гг. (классификация типов — LLM с верификацией экспертом):

| Тип задачи | Доля в корпусе (1 154 задачи) | В тестовом наборе |
|---|:-:|:-:|
| **`modify_entity`** — модификация существующей сущности | **26,0 % (300)** | **15 / 15 (все)** |
| `modify_service` — модификация сервиса | 22,4 % (259) | 0 |
| `bug_fix` — исправление багов | 17,1 % (197) | 0 |
| `add_service` — добавление сервиса / API | 10,0 % (115) | 0 |
| `import_export` — импорт / экспорт | 6,7 % (77) | 0 |
| `add_entity` — добавление новой сущности | 6,7 % (77) | 0 |
| `filtering_sorting` — фильтрация / сортировка | 5,1 % (59) | 0 |
| `ui_view` — UI-представление | 4,7 % (54) | 0 |
| `refactoring` — рефакторинг | 1,0 % (11) | 0 |
| `config_infra` — конфигурация / инфраструктура | 0,4 % (5) | 0 |

`modify_entity` — наибольшая доля среди десяти типов задач (26 % MR-diff). Это согласуется с природой enterprise CRM/ERP-проектов: основная часть инкрементальной разработки — расширение существующих доменных моделей, а не создание сущностей с нуля.

## Два основания концентрации на одном типе

### 1. Самый частый одиночный тип

Покрытие самого частого типа максимизирует репрезентативность тестовой базы для повседневной нагрузки разработки enterprise-приложений. 15 задач × 26 % корпуса = эквивалент покрытия около 4 % всей нагрузки разработки за период 2023–2025.

### 2. Изоляция вклада архитектурных факторов

Концентрация на одном типе задачи **изолирует вклад архитектурных факторов** (мультиагентность, RAG, граф): дисперсия результата между задачами одного типа значительно меньше, чем между задачами разных типов. Это повышает статистическую мощность при выборке *n* = 75 на ячейку и позволяет приписать наблюдаемые различия именно архитектурным факторам, а не вариативности типа задачи.

При смешанном наборе задач (например, 5 `modify_entity` + 5 `add_service` + 5 `bug_fix`) дисперсия between-task увеличилась бы в 2–3 раза (грубая оценка по разнородности файловых паттернов), и эффект архитектурного фактора (например, +10 п.п. от графа) был бы статистически неразличим на той же выборке.

## Ограничение и направление развития

Текущий результат факторного эксперимента **не обобщается** на иные типы задач без отдельной валидации. Перенос на другие типы — задачи R3 (xUnit-сюита T01..T25 — расширение покрытия) и R8 (пилот на стороннем .NET-проекте с гетерогенным набором задач) реестра перспективных задач.
