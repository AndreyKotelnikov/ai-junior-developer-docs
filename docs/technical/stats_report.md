## 4. Методология
- **Wilson 95% CI** для пропорций (`successes/n`); формула Уилсона с z = 1,96.
- **σ Pass@1** — выборочное стандартное отклонение Pass@1 между 5 повторами одного стенда (per-run, ddof = 1): `groupby('run')['pass1'].mean().std(ddof=1)`.
- **FSR (False Success Rate)** — доля наблюдений, где pass1=1 (build OK) И completeness=0 (ни один из эталонных файлов не изменён); строгое определение «формальный успех при полном промахе по адресации».
- **Wilcoxon signed-rank**: парим по 15 задачам — для каждого стенда берём среднее Pass@1 (или Completeness) по всем 5 повторам. Если все разности нулевые, p = 1.
- Используем scipy.stats.wilcoxon с zero_method='wilcox' (нулевые разности отбрасываются), correction=False, alternative='two-sided', method='auto'.
- **Cohen's d_h** для пропорций (Cohen 1988): `d = 2·arcsin(√p₁) − 2·arcsin(√p₀)`.
- **Cohen's d_z** (one-sample) для контраста разностей: `d = mean(diff) / std(diff, ddof=1)`.
- Для **H2b** используется контраст разностей: `(E−D) − (G−B)` per task; нулевая гипотеза — отсутствие синергии RAG×Graph.
- Для **H3** — стек всех разностей по 3 парам (45 наблюдений), one-sample Wilcoxon vs 0.
- Полная методология: `docs/technical/metrics_methodology.md`.

## 5. Робастность вторичных гипотез (compute_robustness_stats.py)

_Источник_: `docs/technical/compute_robustness_stats.py` на тех же 675 наблюдениях; вывод — `stats_output_robustness.json`. Параметры: bootstrap 10 000 ресэмплов, RNG seed = 42. Основные методы: TOST (Schuirmann, 1987), bootstrap (Efron & Tibshirani, 1993), стратификация.

> Скрипт дополнительно вычисляет Bayes factor (JZS) и знаковый тест; эти методы признаны неинформативными (BF попадает в зону неразличимости шкалы Jeffreys 1/3…3; знаковый тест на 9 ненулевых контрастах не обладает разрешающей способностью) и не используются в итоговых выводах.

### 5.1. H2b — направленная формальная поддержка точечной оценки

| Метод | Результат | Формальная мера |

**Вывод по H2b.** Точечная оценка +9,33 п.п. превышает порог +5 п.п.; направление эффекта получает формальную поддержку (bootstrap-вероятность 0,845, монотонная дозозависимость). Вердикт — «условно по точечной оценке»; для строгого формального подтверждения необходимо расширение выборки до n ≈ 184.

### 5.3. Сводный вердикт по семейству пяти гипотез

Дополнительные формальные методы измеряют достоверность направления вторичных гипотез; вердикты по семейству — без изменений: **формально подтверждено 2** (H1, H3); **условно по точечной оценке 2** (H2b, H4); **опровергнута 1** (H2a). Все результаты, включая неблагоприятные, изложены прозрачно (Amrhein, Greenland, McShane 2019).

### 6.1. mean(b·c) на уровне отдельных прогонов

Сравнение формального определения ARR (среднее произведения по 75 прогонам) с агрегатной аппроксимацией (произведение средних).

| Стенд | n | Pass@1 | Completeness | mean(b·c) | Pass@1 × Completeness | Δ |

| H | 75 | 74,67 % | 51,56 % | **40,67 %** | 38,49 % | **+2,17 п.п.** |

Положительный знак Δ подтверждает положительную ковариацию между признаком успешной компиляции `b` и Completeness `c`: успешные прогоны имеют систематически более высокую полноту охвата файлов. Агрегатная формула `Pass@1 × Completeness` даёт **консервативную нижнюю оценку** ARR, занижающую формальное значение `mean(b · c)` на 2–4 п.п. Это согласуется с архитектурной гипотезой: когда система успешно компилирует патч, в среднем она затрагивает большую долю необходимых файлов; провалы сборки чаще связаны с фрагментарной правкой.

### 6.3. Mixed-effects анализ H3 (primary)

Стекированный one-sample Wilcoxon на 45 объединённых разностях для H3 (см. § 2 выше, строка H3: p = 5,7776 × 10⁻⁵) **переоценивает значимость** из-за псевдо-репликации: фактически 45 разностей сняты на одной выборке из 15 задач (3 RAG-условия на каждую задачу) и кластеризованы по задаче. Для корректной инференции применена линейная mixed-effects модель с random intercept на задаче — артефакт `docs/technical/compute_h3_mixed.py` (RNG seed = 42).

**Модель:** `Δ ~ 1 + C(rag) + (1 | task_id)`, метод оценки — maximum likelihood, реализация `statsmodels.MixedLM`.

**Результаты:**

| Метрика | Значение |
|---|:-:|
| Средняя оценка эффекта графа | **+10,56 п.п.** |
| **p-value (mixed-effects, primary)** | **0,000643** |
| Внутрикластерная корреляция (ICC) по задаче | 0,74 |
| Design effect | 2,48 |
| Эффективное n (после поправки на ICC) | ≈ 18 (из 45) |

**Дополнительная проверка робастности.**

| Метод | Результат |
|---|---|
| Cluster-bootstrap по задачам (10 000 ресэмплов) | Средний Δ = +10,62 п.п.; 95 % CI = [+4,58; +16,86] п.п.; P(Δ ≤ 0) = 0,0001 |
| Fisher's combined p по трём раздельным парным Wilcoxon (B→G, C→I, D→E) | p = 0,000463 |

**Согласованность методов.** Все три метода (mixed-effects, cluster-bootstrap, Fisher's combined) дают согласованный вердикт **формального подтверждения** H3 (Δ ≥ порога во всех трёх RAG-условиях; mixed-effects p = 0,000643 < α_adj = 0,01 по Холму; cluster-bootstrap 95 % CI исключает ноль).

**Роль one-sample Wilcoxon (p = 5,7776 × 10⁻⁵, см. § 2).** Используется как **summary-индикатор консистентности** положительного эффекта по всем 45 (task, RAG)-конфигурациям, **а не как primary инферентный тест**. Стекированное p-значение завышено из-за допущения независимости 45 разностей, которое не выполняется (см. ICC = 0,74); mixed-effects p — корректная инферентная оценка с учётом кластеризации.

**Обновление сводной таблицы (§ 2).** Строка H3 интерпретируется как: primary p = **0,000643** (mixed-effects); summary p = 5,7776 × 10⁻⁵ (one-sample Wilcoxon на стеке, без учёта кластеризации). Вердикт **«формально подтверждена»** — по primary mixed-effects.

## Waterfall: декомпозиция шага A → B = +84 п.п. ≠ H1 = +78,67 п.п.

Шаг A → B (стенд A → стенд B) на цепочке через G совмещает **два изменения**:
- архитектура: single → multi;
- RAG: 3 → 0.

Чистый эффект H1 (+78,67 п.п.) измеряется на паре A → C, где RAG = 3 фиксирован на обоих стендах. Арифметическая декомпозиция:

**84,00 = 78,67 (H1, чистый эффект архитектуры) + 5,33 (доп. эффект перехода RAG = 3 → 0 на multi-agent)**

Это объясняет, почему конечный уровень Pass@1 = 90,67 % на стенде E одинаков для обеих цепочек (через D и через G), несмотря на разные промежуточные значения. Конечная точка одна по построению — E — это одна и та же конфигурация: multi + RAG = 5 + граф + Haiku.

### Содержательный результат waterfall — нелинейность путей

Цепочка через D даёт «провал» на шаге B → D (−6,67 п.п., H2a) с компенсирующим скачком D → E (+12,00 п.п., H2b). Цепочка через G монотонна (+2,67 / +2,67). Асимметрия двух путей — эмпирическое доказательство эффекта взаимодействия H2b: добавление RAG поверх multi-agent **без графа** даёт отрицательный вклад, а добавление RAG поверх «multi + граф» — положительный.

## Декомпозиция H1 как открытый исследовательский вопрос (задача R17)

Раздельное измерение вкладов трёх компонентов связки H1 в текущем дизайне 9 стендов **не выполнено**. Для их разделения требуются два дополнительных контрольных стенда:

- **A+** (single + RoslynExplorer-инструмент) — изолирует вклад типизированного семантического анализа;
- **A++** (single + RoslynExplorer + 10 AST-трансформеров) — изолирует вклад детерминированных трансформеров.

### Три компонента связки H1

1. **Ролевая декомпозиция** — разделение работы на Orchestrator → RoslynExplorer → Coder → Tester.
2. **RoslynExplorer 0-LLM-локализация** — типизированный AST-доступ через `Compilation.GetSemanticModel()`.
3. **10 AST-трансформеров в роли Coder** — детерминированные правки типовых EF Core-операций без LLM-вызова.

### Иллюстративная декомпозиция по литературным ориентирам (с дисклеймером)

**Рабочее допущение, формально не доказанное:** литературные диапазоны Python-задач переносятся на enterprise-.NET.

## Архитектурные решения, дающие конкурентоспособность

Полученное превышение ARR(E) = 58,18 % над публикуемыми resolved rate ведущих систем на C# / .NET при на порядок более дешёвой облачной модели объясняется сложением **четырёх независимых архитектурных решений**. Каждое решение по отдельности уже известно в литературе, но вместе в одной системе ранее не применялось.

### 1. Мультиагентная декомпозиция (+78,67 п.п. Pass@1)

Разделение работы на роли Orchestrator → RoslynExplorer → Coder → Tester. Литературные ориентиры на Python: +12 п.п. SWE-Agent, +9 п.п. HyperAgent, +4–5 п.п. MetaGPT. Эффект на enterprise-.NET измерен в гипотезе H1 на связке трёх компонентов (ролевая декомпозиция + типизированная Roslyn-локализация + AST-трансформеры); раздельные вклады компонентов — задача R17 (см. выше).

### 2. Сохранение RAG только в комбинации с графом (опровержение H2a)

Опровержение H2a (Δ = −6,67 п.п. на паре B → D) показало вред RAG без графовой адресации. Производственное решение: режим «RAG без графа» отключён в production; в составе production-конфигурации (стенд E) добавление RAG поверх графа даёт +2,67 п.п. Pass@1 (G → E).

### 3. Граф зависимостей с авторскими COCHANGES-рёбрами (+2,67 п.п. Pass@1 и +10,56 п.п. Completeness)

Граф co-change-зависимостей подсказывает, какие файлы менять вместе. Литературный ориентир графовых систем на Python: +5–8 п.п. (RepoGraph, CodexGraph). Из 16 проанализированных систем кодогенерации co-change-mining как ребро графа зависимостей реализован только в research-прототипах (GraphCoder); промышленной .NET-интеграции в обнаруженной литературе не зафиксировано.

### 4. Синергия графа и RAG (interaction +9,33 п.п. по точечной оценке; H2b)

Граф адресует (где менять), RAG наполняет содержанием (как менять); только комбинация даёт сверхаддитивный эффект. Эффект измерен через разность разностей (E − D) − (G − B) = +9,33 п.п. Маржинальный вклад RAG поверх включённого графа отдельно (пара G → E, +2,67 п.п. Pass@1) уже учтён в пункте 2; здесь не дублируется.

### Арифметика сложения вкладов

Сумма заявленных вкладов не равна A → E = +89,34 п.п. строго по построению — вклады не аддитивны из-за эффекта взаимодействия H2b (+9,33 п.п. interaction). Корректная аддитивная декомпозиция возможна только по одному из двух путей waterfall (см. раздел «Waterfall» выше). Приведённое перечисление — содержательное (вклад каждого решения в отдельности), а не арифметическое.

### Ключевая причина конкурентоспособности — узкая специализация

Внешние системы (SWE-Agent, OpenHands, лидеры SWE-bench) — универсальные: за широту они платят более низким результатом на каждом конкретном классе задач. Настоящая работа намеренно жертвует широтой ради глубины: один тип задач (`modify_entity`) на одном стеке (.NET / EF Core), типизированный анализ кода через Roslyn, детерминированные AST-трансформеры. На канонической приведённой метрике **ARR = 58,18 %** результат превышает публикуемые resolved rate ведущих систем категории прямого сравнения C# / .NET при на порядок более дешёвой облачной модели.
