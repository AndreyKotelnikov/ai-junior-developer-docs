# Заимствованные паттерны, методы и теоретические тезисы (полная версия)

## Назначение

Полная версия таблицы заимствованных паттернов, методов и теоретических тезисов. Здесь приведены развёрнутые формулировки колонки «Авторский вклад в адаптации» с техническими подробностями реализации, цифрами покрытия и инженерными обоснованиями.

Для каждого ключевого источника, из которого в работе берётся не только единичная количественная оценка, но содержательная идея (метод, метрика, теоретический тезис, конструкция), указано: что именно заимствуется и какой авторский вклад внесён в адаптацию.

## Таблица заимствованных паттернов, методов и теоретических тезисов (полная версия)

| Источник | Что именно заимствуется | Авторский вклад в адаптации |
|---|---|---|
| MetaGPT | SOP-driven декомпозиция; передача артефактов между ролями | Замена «мягкой» SOP-координации на **жёсткие программные gate'ы**: переход к следующей фазе пайплайна разрешён только при успешном результате детерминированной проверки — `dotnet build` (компилируемость) и регрессионная xUnit-сюита (функциональная корректность); вместо подписанного человеком SOP-документа — машинно-проверяемый бинарный сигнал {pass / fail}, не зависящий от стохастичности LLM. **4 роли вместо 5**: вместо стандартной пятёрки MetaGPT (Product Manager + Architect + Project Manager + Engineer + QA) в эксперименте оставлены **Orchestrator** (план задач), **RoslynExplorer** (семантическая локализация без LLM-вызовов), **Coder** (генерация diff) и **Tester** (запуск gate'ов и анализ ошибок); функции Product Manager и Architect сжаты в Orchestrator, Project Manager заменён программным оркестратором LangGraph |
| SWE-Agent | Tester и цикл сборки (action-observation loop) | Замена «слепой» LLM-петли (агент пробует — наблюдает stderr — пробует снова, исключительно через LLM) на **гибридный цикл с детерминированной первой линией**: 10 ProgrammaticTransformers покрывают типовые EF Core-операции без LLM-вызова; 5 фиксеров типовых CS-ошибок (CS0246 «не найден тип», CS0117 «нет члена с таким именем», CS1061 «отсутствует определение метода», CS0535 «не реализован член интерфейса», CS0738 «несовпадение возвращаемого типа») исправляют около 60 % ошибок программно; LLM-вызов остаётся только для остатка нетривиальных случаев — экономия токенов 7 K → 2,5 K на подзадачу |
| MapCoder | Четырёхагентная архитектура | Замена LLM-агента **Retrieval** (в MapCoder он ищет похожие задачи через эмбеддинги — стохастично и зависит от качества базы примеров) на **детерминированный микросервис `roslyn-graph`** (.NET 9, около 50 эндпоинтов OpenAPI): семантический граф C# с четырьмя типами рёбер CALLS / INHERITS / USES_TYPE / COCHANGES; обход графа выдаёт точный, воспроизводимый результат локализации |
| HyperAgent | Per-agent permissions | Реализация per-agent ограничений как **middleware-политики FastAPI**: разрешённый набор инструментов (`tools_whitelist`) и операций fs / docker / network для каждого агента описан декларативно в YAML и проверяется на уровне HTTP-роутера до выполнения — нет возможности «обойти» политику через промпт-инъекцию; HyperAgent описывает идею, а реализация на уровне http запроса от веб-фреймворка — собственный инженерный вклад |
| AutoGen v0.4 | Actor-model, оптимальный диапазон 5–7 ролей | На периметре факторного эксперимента сознательно использованы **4 роли** (Orchestrator + RoslynExplorer + Coder + Tester) — чтобы изолировать вклад мультиагентности от вклада дополнительных «вспомогательных» ролей; в production-конфигурацию (вне факторного дизайна) добавлены ещё 3 роли: **Slim Coder** (компактный кодер с prompt caching), **FixAgent** (LLM-фиксер нестандартных CS-ошибок) и **ExplainerAgent** (post-hoc объяснение в UI) — достигая верхней границы оптимального диапазона 5–7 |
| ChatDev | Каскадная мультиагентная модель (waterfall) | **Sequential-pipeline отвергнут** в пользу LangGraph-DAG (DAG — directed acyclic graph, направленный ациклический граф): waterfall блокирует параллелизацию подзадач и приводит к деградации при ошибке любой фазы; DAG позволяет параллельно обрабатывать независимые подзадачи и выборочно повторять только сбойный узел. |
| Agentless | Принцип «локализация → починка» как анти-агентный baseline | Применяем тот же двухфазный принцип, но **на классе типовых EF Core-задач** — там, где LLM избыточна: для добавления навигационного свойства, регистрации ассоциации в `DataConfiguration.cs` и подобных операций со строго определённым синтаксисом выполняется детерминированный AST-трансформер без LLM-вызова (на эталонной задаче — 12 из 13 подзадач, около 92 %) |
| Self-Organized Agents | Тезис «мультиагентная декомпозиция превосходит сильный одноагентный baseline» | Перенос тезиса **с HumanEval (+5 п.п. в литературе)** на enterprise EF Core с более строгим порогом **+10 п.п.**: enterprise-задачи затрагивают 5–7 файлов одновременно и упираются в позиционную деградацию контекста одноагентной архитектуры — отсюда ожидание большего по абсолютной величине эффекта; гипотеза проверяется отдельным стендом сравнения single vs multi (см. H1) |
| GraphCoder / CGM | Graph-augmented retrieval; тезис о сверхаддитивности RAG × граф | **Compiler-level семантический граф** через Roslyn (точные ссылки символов, биндинг типов) вместо AST / текстового графа (синтаксические узлы без семантики, как в GraphCoder); рёбра **COCHANGES** на основе истории git-cochange (FP-Growth на корпусе 1 154 пар MR-diff) — **самостоятельный научно-инженерный вклад работы**, отсутствующий у GraphCoder и CGM |
| FP-Growth (Han et al., SIGMOD 2000) | Алгоритм извлечения частых ассоциативных правил | Применение алгоритма из транзакционного майнинга к **корпусу MR-diff корпоративного GitLab** (1 154 пары «требование → код») с настройкой порогов `min_support = 0,05` (правило встречается в ≥ 5 % MR), `min_lift = 1,2` (пара файлов меняется вместе в ≥ 1,2 раза чаще, чем при случайной совместности); полученные 84 co-change-правила интегрированы как **типизированные рёбра** в семантический граф Roslyn |
| Zimmermann TSE 2005 | Фундаментальный тезис: история совместных изменений предсказывает релевантные файлы | Перенос идеи **с open-source Eclipse / Mozilla** на enterprise .NET / EF Core; интеграция в **семантический** граф как типизированного ребра COCHANGES — в отличие от Zimmermann, где правила использовались как самостоятельная рекомендательная система; live-эндпоинт `/api/cochange/files` (cold ≈ 900–1 400 мс, warm < 200 мс compute) — собственная реализация |
| CodeRAG-Bench | Тезис «качество retrieval > количество примеров»; эмпирический разброс эффекта от −10 до +35 п.п. | **Прогноз отрицательной дельты RAG в изоляции** для узкоспециализированного корпуса EF Core при низком качестве retrieval-движка — этот прогноз закрепляется как гипотеза H2a и проверяется отдельным стендом (RAG = 5, граф = off) |
| Lost in the Middle (Liu N. F. et al., TACL 2024) | Тезис о позиционной деградации внимания при контексте более 6 K токенов | Применение бюджета **6 K токенов как методологического ограничения** локальной развёртки (`OLLAMA_NUM_CTX = 6144` для `qwen2.5-coder:14b`) — против стандартных 12 K из коробки Ollama; обоснование разделения промпта Coder на **статический префикс** (около 3,4 K токенов, кэшируется через `cache_control: ephemeral`) **+ динамический per-subtask блок** (около 1 K токенов) в Slim Coder; снижение объёма промпта на 70 % |
| Qwen2.5-Coder Tech Report | Характеристики базовой open-source-модели (5,5 трлн токенов; 73,7 % Aider 32B) | Выбор конкретной квантизации **Q4_K_M** как нижней границы 16 ГБ VRAM (RTX 3080 Ti Laptop) — компромисс между размером модели в видеопамяти (≈ 9 ГБ) и качеством генерации; модель отобрана из 8 кандидатов (`codellama`, `deepseek-coder`, `starcoder2`, `granite-code` и др.) — полная таблица сравнения вынесена в `docs/technical/ollama_model_research_full.md` |
| Mukherjee R. et al. *Neural Program Generation Modulo Static Analysis* (NeurIPS 2021) | Тезис: чисто нейросетевая генерация неизбежно даёт семантические ошибки; требуется присоединение статанализа | На эталонной задаче «доработка существующей сущности» (12 из 13 подзадач без LLM, около 92 %) детерминированный путь покрывает большинство подзадач; для прочих типов задач подробных замеров не проводилось — повышение этого показателя является направлением дальнейшей работы |
| Hybrid-Code v2 | Жизнеспособность гибрида детерминированной верификации и LLM с низким уровнем галлюцинаций | Применение нейросимвольной парадигмы **к узкому классу типовых EF Core-операций** (а не к произвольной кодогенерации, как в Hybrid-Code v2): 10 параметризованных AST-трансформеров + 5 классов ProgrammaticFixer покрывают около 92 % подзадач эталонной задачи без LLM; LLM остаётся только для нестандартных случаев — экономия LLM-вызовов 5 → 1 на подзадачу |
| AWS re:Invent 2025 (Byron Cook) + WEF декабрь 2025 | Индустриальное и институциональное подтверждение магистральности нейросимвольного направления | Использование как **контекстного аргумента актуальности направления** на конец проекта (май 2026 г.): два независимых источника — Byron Cook (Distinguished Engineer AWS, отвечающий за formal verification) и Всемирный экономический форум — фиксируют нейросимвольный гибрид как магистральное направление 2025–2027 гг.; работа позиционируется как **ранняя индустриальная реализация** парадигмы в регулируемом сегменте РФ |
| Khatchadourian *Replayable Financial Agents* | Тезис «No model achieves both perfect determinism and high accuracy» (4 700+ агентских прогонов) | Применение тезиса как **обоснования двух взаимосвязанных архитектурных решений**: (а) **локальная развёртка** через Ollama с `temperature = 0` как стратегия максимизации воспроизводимости (целевой ориентир 100 % byte-identical replay — задача R10); (б) **усреднение по 5 повторам** на каждой ячейке факторного эксперимента как стратегия контроля остаточной стохастичности облачных провайдеров (Claude Haiku 4.5 даёт 50 % byte-identical при `temperature = 0`) |
| R-LAM (arXiv:2601.09749, январь 2026) | Концепция provenance-tracking и детерминированных политик исполнения | Реализация фиксации каждого LLM-вызова в Postgres-таблице с REVOKE на DML (отзыв прав на команды класса Data Manipulation Language) |
| SWE-Bench Pro (Deng X. et al., 2025) | Литературный ориентир 30–50 % паритета для on-prem 14B-моделей | Установление порога 75 % как амбициозной цели на нижней границе 16 ГБ VRAM. Существенное повышение порога планируем достичь за счёт использования следующих инструментов: Roslyn-семантический граф, Гибрид Det + LLM, Per-agent ModelRouter (каждый агент получает лучший локальный инструмент) |
| Coming (Martinez & Monperrus) | Демонстрация наличия research-only co-change-инструментов без промышленной интеграции | Промышленная интеграция co-change-рёбер в Roslyn Graph как самостоятельный вклад |
