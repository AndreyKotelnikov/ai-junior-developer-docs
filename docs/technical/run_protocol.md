# Протокол одного прогона факторного эксперимента

Каждый из 675 прогонов (9 стендов × 5 повторов × 15 задач) выполняется по единому протоколу.

## 9 шагов прогона

1. Бизнес-требование подаётся в Orchestrator в формате `business_requirement.md`.
2. Orchestrator декомпозирует требование на подзадачи и формирует план.
3. RoslynExplorer строит граф зависимостей для целевых сущностей (0 LLM-вызовов).
4. Для каждой подзадачи Coder генерирует C#-код в формате EDIT-block.
5. Tester запускает `docker build`, парсит CS-ошибки.
6. При CS-ошибках срабатывает fix-loop (максимум 3 итерации Coder ↔ Tester).
7. Замеряется TimeToSuccess — время от подачи требования до возврата успешного diff.
8. Множество F_actual фактически изменённых файлов сопоставляется с эталонным F_required (зафиксирован до прогонов), вычисляется Completeness = |F_actual ∩ F_required| / |F_required|.
9. Прогон фиксируется как `result_<run_id>.json` в `docs/experiment/test_results/stand-<id>-run-<n>.md`.

## Параметры стандартного прогона

- `temperature = 0` на всех 9 стендах.
- Серийный планировщик Ollama для on-prem-стендов (одна видеокарта 16 ГБ VRAM).
- Конфигурации факторов (Architecture / RAG / Graph / Model) — `docs/experiment/stand_configs/docker-compose.stand-{a..i}.yml`.
- Тестовый набор 15 задач — `docs/experiment/0.1_tests.md`.

## Время выполнения

- Один стенд (5 повторов × 15 задач = 75 прогонов): ~ 12 часов на cloud-стенд; ~ 24 часа на on-prem-стенд.
- Полный эксперимент (9 стендов × 75 прогонов = 675 наблюдений): ~ 150 часов с учётом параллельного запуска cloud-стендов.
