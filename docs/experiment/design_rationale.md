# Принципы отбора 9 практических конфигураций и технический каркас стендов

Документ обосновывает структуру дизайна факторного эксперимента: отбор 9 практических конфигураций из 24 ячеек полного декартова произведения четырёх факторов и технические параметры стендов A–I.

## Принципы отбора 9 практических конфигураций

Дизайн **неполный факторный**. Полное декартово произведение четырёх факторов (Architecture {single, multi} × RAG {0, 3, 5} × Graph {off, on} × Model {Haiku, Ollama}) содержит **24 ячейки**. Стенд A фиксирует единственную точку single-agent (baseline для проверки H1); остальные 8 стендов B–I — 8 из 12 multi-agent-конфигураций (3 уровня RAG × 2 значения Graph × 2 модели).

4 неиспользованные multi-конфигурации избыточны или ресурсно недоступны:

1. **Ollama + RAG = 5 + graph = off** — превышает бюджет контекста локальной модели 6 K токенов (квантизация Q4_K_M на 16 ГБ VRAM); RAG = 5 даёт ~ 8 K токенов контекста для Coder, что не помещается.
2. **Ollama + RAG = 5 + graph = on** — то же ограничение контекста, что и в (1).
3. **Ollama + RAG = 3 + graph = on** — для on-prem-конфигурации с RAG = 3 не даёт нового парного сравнения сверх уже зафиксированных в стендах F (RAG = 3, graph = off, Ollama) и H (RAG = 0, graph = on, Ollama); парность по фактору графа уже изолирована в C ↔ I при Haiku.
4. **Ollama + RAG = 0 + graph = off** — исключена сознательно для сохранения парности G ↔ H (граф = on) как основной для H4; при сохранении этой пары добавление чисто on-prem-baseline без графа требовало бы дополнительной серийной серии прогонов локальной модели, не дающей новых пар сравнения для H4.

Пространство дизайна сокращено по трём принципам:

1. **Единственный single-agent baseline.** Архитектура `single` нужна в одной точке для проверки H1 (стенд A); во всех остальных стендах архитектура фиксирована как `multi`. Сочетание одноагентной архитектуры с дополнительными факторами не даёт нового парного сравнения.
2. **Фиксация модели на cloud (Claude Haiku 4.5) для всех гипотез, кроме H4.** Гипотезы H1, H2a, H2b, H3 проверяются на стабильной cloud-модели для устранения шума локального инференса; локальная модель `qwen2.5-coder:14b` присутствует только в стендах F и H (тестовая пара для H4).
3. **Парные стенды с минимальной разностью.** Каждый стенд имеет хотя бы один парный, отличающийся ровно одним фактором, — это позволяет приписать наблюдаемый эффект конкретному фактору (например, B vs D — отличие только в RAG; B vs G — только в Graph).

Роль ключевых стендов:

- **A** — единственный single-agent (контроль H1, Pass@1 = 1,33 %);
- **B** — нулевая точка multi-agent baseline для H2a и H3;
- **E** — референсная облачная конфигурация (лидер Pass@1 = 90,67 %);
- **G** — лидер Completeness (62,44 %), рекомендованная cloud production-конфигурация;
- **H** — главная конфигурация air-gapped-контура для регулируемого сегмента (паритет 84,85 % с G при 16 ГБ VRAM, квантизация Q4_K_M).

## Технический каркас стендов

Каждый стенд запускается отдельным compose-файлом `docker-compose.stand-<id>.yml`. Общий каркас:

- **multiagent** — FastAPI-сервис, центр пайплайна;
- **rag-service** — Qdrant-индекс корпуса MR-diff (для стендов с RAG > 0);
- **roslyn-graph** — детерминированный анализатор зависимостей (для стендов с `graph = on`);
- **qdrant** — векторная БД для RAG-поиска;
- **postgres** — `pipeline_audit` (append-only журнал прогонов);
- **ollama** — локальная LLM-инфраструктура (только на стендах F и H).

Параметры инференса:

- `temperature = 0` на всех стендах (как cloud Haiku, так и локальной Ollama);
- параметр `seed` в текущей реализации Ollama явно не фиксируется (ограничение воспроизводимости — задача R10 реестра задач);
- max_tokens Orchestrator = 16384 (поддержка планов из 34+ подзадач);
- max_tokens Coder / Tester = по умолчанию модели.

Воспроизводимость обеспечена двумя артефактами:

1. Git-репозиторий с зафиксированным коммитом;
2. 9 Docker Compose-конфигураций с явной фиксацией значений факторов (`docs/experiment/stand_configs/docker-compose.stand-{a..i}.yml`).

Эти артефакты в совокупности позволяют любому эксперту повторить факторный прогон 9 × 5 × 15 = 675 наблюдений.
