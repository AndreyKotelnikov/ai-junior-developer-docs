# Сценарий Б — Compliance-офицер: полное описание

**Назначение**: пошагово описать end-to-end сценарий работы compliance-офицера — от поиска прогона по `run_id` до выгрузки `audit-{run_id}.json` для регулятора с побайтовым воспроизведением LLM-вызовов.
**Связанные документы**:
- [`../compliance/compliance-matrix.md`](../compliance/compliance-matrix.md) — соответствие 152-ФЗ / 187-ФЗ / ФСТЭК.
- [`cli_replay_manual.md`](cli_replay_manual.md) — руководство по CLI побайтового воспроизведения.
- [`scenario_a_business_user_full.md`](scenario_a_business_user_full.md) — сценарий бизнес-пользователя (источник `run_id`).

> Сценарий Б — второй из трёх непересекающихся пользовательских сценариев MVP. Точка входа — страница `/compliance`; узловая точка — `ReplayPanel` (byte-identical replay); артефакт результата — `audit-{run_id}.json` для регулятора. Целевая длительность — ≈ 3 минуты.

---

## 1. Роль и предусловия

| Параметр | Значение |
|---|---|
| Роль | Compliance-офицер банка / КИИ-оператора (152-ФЗ, 187-ФЗ, Приказы ФСТЭК № 235 / 239) |
| Точка входа | страница `/compliance` веб-интерфейса |
| Предусловия | существует завершённый прогон с известным `run_id` (результат Сценария А) |
| Узловая точка | `ReplayPanel` — побайтовое воспроизведение LLM-вызовов прогона |
| Артефакт результата | `audit-{run_id}.json` — пакет для предъявления регулятору |
| Целевая длительность | ≈ 3 минуты |

---

## 2. Шаги сценария

### Шаг 1. Открыть страницу `/compliance`

Compliance-офицер открывает страницу `/compliance` — панель аудита прогонов. Страница показывает журнал `pipeline_audit` (append-only) с записями LLM-вызовов по всем прогонам.

### Шаг 2. Найти прогон по `run_id`

Офицер вводит `run_id` (получен от бизнес-пользователя или из реестра задач). Записи фильтруются по `run_id`; видны `sequence`, `agent_name`, `model_name`, `prompt_hash` / `completion_hash` (SHA-256), `timestamp_utc`, `duration_ms`.

### Шаг 3. Открыть `ReplayPanel`

`ReplayPanel` отображает все LLM-вызовы выбранного прогона в порядке `sequence` с полным контекстом: `full_prompt`, `full_completion`, `git_commit_before/after`, `subtask_id`, `subtask_why`.

### Шаг 4. Запросить byte-identical replay

Офицер запускает повторное воспроизведение прогона. Воспроизводимость зависит от провайдера:

| Провайдер | Byte-identical replay | Причина |
|---|---|---|
| Claude Haiku 4.5 (Anthropic) | 2 из 4 (50 %) | ограничение `temperature` на стороне cloud-провайдера |
| Ollama (локально) + фиксация `seed` | плановое 100 % | задача R10 — фиксация параметров инференса |

Хеши вычисляются как SHA-256 от детерминированной сериализации сообщений: `json.dumps(messages, sort_keys=True, ensure_ascii=False)`.

### Шаг 5. Выгрузить `audit-{run_id}.json`

Офицер выгружает пакет `audit-{run_id}.json` — машиночитаемый артефакт со всеми записями `pipeline_audit` по прогону, хешами и временными метками. Пакет предъявляется регулятору и поддаётся независимой проверке без доступа к исходному коду агентов.

## 3. Инструменты проверки

| Инструмент | Назначение |
|---|---|
| CLI `python -m app.cli.multiagent_replay --run-id=<UUID>` | побайтовое воспроизведение прогона из командной строки |
| REST `GET /api/audit/runs/{run_id}` | программный доступ к записям прогона |
| `psql` команда `\dp pipeline_audit` | проверка привилегий роли `multiagent_app` (только INSERT / SELECT) |
| `ots verify <hash>.ots` | проверка опциональной нотаризации OpenTimestamps |

Полный compliance-drill — 3 bash-команды, ≈ 5 минут (см. `../compliance/compliance-matrix.md`, разд. 3).

---

## 4. Связь с нормативной базой

| Требование | Механизм сценария |
|---|---|
| 152-ФЗ ст. 18.1 — учёт операций с ПДн | append-only `pipeline_audit`, выборка по `run_id` |
| 152-ФЗ ст. 18.1 ч. 2 п. 3 — оценка эффективности мер | 5-минутный compliance-drill, byte-identical replay |
| Приказ ФСТЭК № 235 п. 4.2 — контроль целостности | SHA-256 `prompt_hash` / `completion_hash`; OpenTimestamps |
| Указ Президента РФ № 490 — контроль человека над ИИ | `subtask_why` каждого решения Orchestrator в журнале |

---

## 5. Известные ограничения

- `full_prompt` и `full_completion` хранятся в открытом виде — шифрование на уровне СУБД (TDE / AES-GCM) вынесено в задачу R14.
- 100 % byte-identical replay для Ollama — плановое значение, достигается при фиксации `seed` (задача R10).
- Формальная архивная retention сроком не менее 5 лет — формализация процедуры выгрузки вынесена в задачу R15.
