# Сценарий В — Администратор онбординга: полное описание

**Назначение**: пошагово описать end-to-end сценарий подключения нового .NET-проекта к системе — от запуска авто-обнаружения конвенций до активации `project_conventions.json` с горячей перезагрузкой.
**Связанные документы**:
- [`scenario_a_business_user_full.md`](scenario_a_business_user_full.md) — сценарий бизнес-пользователя.
- [`scenario_b_compliance_officer_full.md`](scenario_b_compliance_officer_full.md) — сценарий compliance-офицера.

> Сценарий В — третий из трёх непересекающихся пользовательских сценариев MVP. Точка входа — страница `/onboarding`; узловая точка — шаг `review` (таблица `FieldRow`); артефакт результата — `project_conventions.json` + горячая перезагрузка. Целевая длительность — ≈ 4 часа (против ≈ 3 дней ручного анализа — ускорение ≈ ×18).

---

## 1. Роль и предусловия

| Параметр | Значение |
|---|---|
| Роль | Администратор онбординга — администратор ИТ-департамента, подключающий новый .NET-проект |
| Точка входа | страница `/onboarding` веб-интерфейса (4-шаговый мастер) |
| Предусловия | исходный код нового .NET-проекта доступен системе |
| Узловая точка | шаг `review` — человек проверяет draft-конвенции в таблице `FieldRow` |
| Артефакт результата | `project_conventions.json` + горячая перезагрузка через `POST /api/conventions/load` |
| Целевая длительность | ≈ 4 часа (ручной аналог — ≈ 3 дня) |

---

## 2. Шаги сценария

### Шаг 1. Открыть мастер `/onboarding`

Администратор открывает 4-шаговый мастер на странице `/onboarding` и указывает целевой .NET-проект.

### Шаг 2. Авто-обнаружение конвенций

Запускается конвейер авто-обнаружения на пяти фокус-детекторах Roslyn-графа:

| Фокус-детектор | Что извлекает |
|---|---|
| `AstPatternMiner` | AST-паттерны кода |
| `InitializerDetector` | паттерны регистрации в `Initializer` |
| `ExternalPackageDetector` | префиксы внешних пакетов (`Visary.`, `Microsoft.`, …) |
| `EntityBaseTypeDetector` | базовые типы сущностей |
| `RegistrationPatternMiner` | шаблоны регистрации ассоциаций |

Прохождение по кодовой базе занимает ≈ 5 секунд (на эталонном `dotnet-app` — 1 559 .cs-файлов); recall авто-обнаружения — **87,6 %** (валидировано на `dotnet-app`). Результат — draft-документ `project_conventions.json` (≈ 180 строк).

### Шаг 3. Review draft (узловая точка)

Draft-конвенции предъявляются администратору в таблице `FieldRow` — построчное представление полей `project_conventions.json`: `project_id`, `language`, `file_patterns`, `registration_patterns` (≈ 11 шаблонов), `external_package_prefixes`, `known_external_types`, `entity_base_types`, `operations`. Администратор проверяет и при необходимости корректирует значения. Авто-обнаружение **не применяется автоматически** — финальная редакция остаётся за человеком.

### Шаг 4. Сохранить и активировать

После утверждения документ валидируется в две ступени:
1. структурно — JSON Schema 2020-12 (jsonschema-валидатор);
2. типово — Pydantic v2 с `extra="forbid"` (5 схем).

Валидный документ активируется горячей перезагрузкой через `POST /api/conventions/load` — без рестарта сервиса. С этого момента новый проект доступен для Сценария А.

## 3. Артефакт результата

| Поле `project_conventions.json` | Назначение |
|---|---|
| `project_id` | идентификатор проекта |
| `language` | язык целевого кода |
| `file_patterns` | маски расположения файлов |
| `registration_patterns` | ≈ 11 шаблонов регистрации ассоциаций |
| `external_package_prefixes` | префиксы NuGet-пакетов вне проекта |
| `known_external_types` | известные внешние типы |
| `entity_base_types` | базовые типы сущностей EF Core |
| `operations` | поддерживаемые типы операций |

Тестовое покрытие схемы — 64 unit-теста.

## 5. Известные ограничения

- Recall 87,6 % означает, что ≈ 12 % конвенций требуют ручного дополнения на шаге `review`.
- Авто-обнаружение валидировано на `dotnet-app`; на проектах иной структуры recall может отличаться.
- Подключение проектов масштаба ≥ 5 000 файлов вынесено в задачу роста R8.
