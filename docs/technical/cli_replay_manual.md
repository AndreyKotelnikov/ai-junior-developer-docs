# Мануал CLI `multiagent_replay`

`multiagent_replay` — инструмент побайтового воспроизведения записанного прогона
пайплайна из журнала аудита `pipeline_audit`. Для каждого записанного LLM-вызова
исходный промпт повторно отправляется той же модели, а SHA-256-хеш полученного
ответа сравнивается с сохранённым. На детерминированном бэкенде (`temperature = 0`)
ожидается побайтовое совпадение; расхождения помечаются по шагам, и код возврата
становится ненулевым, чтобы их можно было отследить автоматически.

Запуск (изнутри контейнера `multiagent`):

```bash
python -m app.cli.multiagent_replay --run-id <UUID>
```

## 1. Опции командной строки

Поддерживаются шесть опций.

| Опция | Назначение |
|---|---|
| `--run-id <UUID>` | UUID прогона для воспроизведения. Обязательна, если не указана `--list`. |
| `--list` | Вывести последние `--limit` идентификаторов прогонов и выйти. |
| `--limit N` | Сколько `run_id` выводить при `--list` (по умолчанию 50). |
| `--no-call` | Не выполнять LLM-вызовы: напечатать только манифест прогона из журнала (полезно для инспекции без расхода токенов). |
| `--json` | Выводить результат в формате JSON вместо человекочитаемой таблицы. |
| `--ignore-bundle-mismatch` | Продолжить, даже если активный документ конвенций `project_conventions` отличается от записанного с прогоном. Без этого флага реплей прерывается с кодом 3. |

## 2. Коды возврата

| Код | Значение |
|:-:|---|
| 0 | Успех: все шаги byte-identical, либо успешно выведен манифест/список. |
| 1 | Есть расхождения при реплее **или** прогон с указанным `run_id` не найден. |
| 2 | Журнал аудита не сконфигурирован (`AUDIT_DATABASE_URL`) или не передан `--run-id`. |
| 3 | Активный bundle конвенций отличается от записанного (реплей разошёлся бы); переопределяется флагом `--ignore-bundle-mismatch`. |

## 3. Что делает инструмент по шагам

1. Получает все записи прогона из `pipeline_audit` по `run_id`, упорядоченные по
   полю `sequence`.
2. Проверяет дрейф bundle: сравнивает `bundle_sha256` записанного прогона с
   текущим активным. При несовпадении и без `--ignore-bundle-mismatch`
   завершается с кодом 3.
3. Печатает манифест прогона: число LLM-вызовов, время начала/окончания (UTC),
   базовый коммит (`git_commit_before`), затем по каждому шагу — номер,
   агент, модель, число токенов, длительность (мс).
4. Если не задан `--no-call`: для каждой записи восстанавливает сообщения из
   `full_prompt`, выбирает модель того же агента (учитывая per-agent-маршрутизацию)
   и повторно вызывает её с теми же параметрами (`temperature`, `max_tokens`,
   `json_mode` из `metadata`). SHA-256 полученного ответа сравнивается с
   сохранённым `completion_hash`.
5. Печатает итог: `N/M byte-identical, K drifted` и пометку `OK`/`MISMATCH` по
   каждому шагу. Код возврата 0, если расхождений нет, иначе 1.

## 4. Важные ограничения

- **Инструмент не восстанавливает рабочее дерево** `dotnet-app/src`. Журнал
  фиксирует базовый коммит (`git_commit_before` / `git_commit_after`) для
  трассируемости, но восстановление состояния репозитория (обычно
  `git checkout <baseline>`) выполняется оператором отдельно.
- Чтобы повторные вызовы реплея сами не записывались в журнал аудита,
  воспроизведение использует базовый LLM-клиент (а не аудитирующий).
- Побайтовое совпадение достигается только на детерминированном бэкенде. На
  облачном провайдере даже при `temperature = 0` детерминизм не гарантируется
  (фактически — 2 из 4 прогонов byte-identical); на локальном бэкенде Ollama
  параметр `seed` в текущей реализации не фиксируется, поэтому 100 %-е
  воспроизведение — плановая величина (задача R10).

## 5. Примеры

Список последних 20 прогонов:

```bash
python -m app.cli.multiagent_replay --list --limit 20
```

Инспекция манифеста без вызова моделей:

```bash
python -m app.cli.multiagent_replay --run-id 4f1c… --no-call
```

Полное воспроизведение с выводом в JSON:

```bash
python -m app.cli.multiagent_replay --run-id 4f1c… --json
```

Воспроизведение при изменившемся документе конвенций:

```bash
python -m app.cli.multiagent_replay --run-id 4f1c… --ignore-bundle-mismatch
```

Программный доступ к записям прогона — REST-эндпоинт `GET /api/audit/runs/{run_id}`.

## 6. Протокол операционной воспроизводимости результатов

Любой ключевой числовой результат воспроизводится за четыре шага без обращения к
авторам:

1. Получить репозиторий на фиксированном коммите: `git checkout <SHA>`.
2. Развернуть нужный стенд X ∈ {A…I}:
   `docker compose -f docs/experiment/stand_configs/docker-compose.stand-<X>.yml up`.
3. Запустить факторный прогон (5 повторов × 15 задач):
   `python docs/experiment/hyp-run-tests.py --stand <X>` → 75 результатов в
   `docs/experiment/test_results/stand-<X>-run-<n>.md`.
4. Пересчитать метрики и статистику: `python docs/technical/compute_stats.py` →
   отчёт `docs/technical/stats_report.md` и сводка `docs/technical/stats_output.json`.

Ориентир по времени: один cloud-стенд ≈ 12 ч, один on-prem-стенд ≈ 24 ч; полный
эксперимент (9 стендов × 75 прогонов = 675 наблюдений) ≈ 150 ч при параллельном
запуске cloud-стендов.
