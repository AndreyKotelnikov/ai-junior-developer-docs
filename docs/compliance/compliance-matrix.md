# Матрица нормативного соответствия

**Назначение**: показать, как технические возможности системы AI Junior Developer
(append-only-журнал `pipeline_audit`, ограниченная роль `multiagent_app`,
CLI побайтового воспроизведения и опциональная нотаризация OpenTimestamps)
закрывают требования ключевых регуляторов российского законодательства:
**152-ФЗ** «О персональных данных», **187-ФЗ** «О безопасности критической
информационной инфраструктуры», Приказы **ФСТЭК № 235** и **№ 239**, а также
требование контроля человека над ИИ (**Указ Президента РФ № 490**).

Каждый пункт матрицы проверяется SQL-запросом или CLI-командой. Статус
реализации обозначен так: **ТР** — техническая возможность реализована (technically ready),
**ТГ** — система технологически готова, требуется организационное оформление субъектом (technically guaranteed/pending).

## 1. Сводная матрица требований

| № | Норматив | Требование | Механизм в системе | Контрольная проверка | Статус |
|:-:|---|---|---|---|:-:|
| 1 | 152-ФЗ, ст. 19 ч. 2 п. 9 | Учёт обращений к информационной системе персональных данных | Каждый LLM-вызов пишется в `pipeline_audit` (`run_id`, `sequence`, `agent_name`, `model_name`, `prompt_hash`, `completion_hash`, число токенов, метка времени) | `SELECT count(*) FROM pipeline_audit WHERE run_id=$1` | ТР |
| 2 | 152-ФЗ, ст. 19 ч. 2 п. 5 | Неизменность журналов обращений | Роль `multiagent_app` имеет только `INSERT, SELECT`; `UPDATE`, `DELETE`, `TRUNCATE` явно `REVOKE`-нуты | `psql -U multiagent_app -c "DELETE …"` → permission denied | ТР |
| 3 | 152-ФЗ, ст. 22.1 ч. 4 | Контроль передачи данных третьим лицам (внешний LLM) | Поле `model_name` фиксирует провайдера: в режиме API — `anthropic/…`, в режиме Ollama — локальный путь | `SELECT DISTINCT model_name FROM pipeline_audit` | ТР |
| 4 | Постановление Правительства РФ № 1119/2012, п. 14 | Контроль действий пользователей и автоматизированных средств | Поля `agent_name`, `subtask_id`, `subtask_why`, `git_commit_before/after` дают полную трассу «кто — что — зачем» | `SELECT agent_name, subtask_why FROM pipeline_audit WHERE run_id=$1` | ТР |
| 5 | 187-ФЗ, ст. 9 п. 3 | Обнаружение и реагирование на инциденты | На каждый LLM-вызов фиксируется `duration_ms` и `git_commit_before/after`; аномальная задержка или неожиданное изменение репозитория видны в журнале | `SELECT duration_ms FROM pipeline_audit WHERE run_id=$1` | ТР |
| 6 | Приказ ФСТЭК № 235, разд. II.2 | Архивация событий безопасности не менее 5 лет | `pipeline_audit` без TTL; сжатие на уровне tablespace PostgreSQL | `SELECT min(ts) FROM pipeline_audit` | ТГ |
| 7 | Приказ ФСТЭК № 239, разд. VI.1 | Проверка целостности средств защиты | SHA-256 `prompt_hash`/`completion_hash` плюс опциональная нотаризация (OpenTimestamps) доказывают, что журнал не менялся постфактум | `ots verify <hash>.ots` | ТР |
| 8 | Приказ ФСТЭК № 239, разд. V | Разграничение доступа | Роль `multiagent_app` ограничена `INSERT/SELECT`; административные операции — отдельная роль `postgres` | `\dp pipeline_audit` | ТР |
| 9 | Приказ ФСТЭК № 239 (изоляция значимых объектов) | Изоляция контура для объектов КИИ высших категорий | Air-gapped-режим: все LLM-вычисления локальны (Ollama + локальный PostgreSQL), исходящий HTTP-трафик отсутствует | `curl --max-time 5 https://api.anthropic.com` → connection refused | ТР |
| 10 | Приказ ФСТЭК № 235 (организация защиты) | Документирование и категорирование объекта КИИ | Подготовлен акт категорирования (III категория значимости) | см. [`categorisation_act.md`](categorisation_act.md) | ТГ |
| 11 | Указ Президента РФ № 490 | Контроль человека над искусственным интеллектом | Узел `ApprovalGate` требует явного утверждения плана человеком до кодирования; решение фиксируется в `subtask_why` | UI-блок `ApprovalGate`; поле `pipeline_audit.subtask_why` | ТР |
| 12 | 187-ФЗ, ст. 7 | Категорирование значимых объектов КИИ и передача сведений во ФСТЭК | Акт категорирования (III категория) готов; формальная регистрация в реестре ФСТЭК — задача R15 | см. [`categorisation_act.md`](categorisation_act.md), разд. 5 | ТГ |
| 13 | 152-ФЗ / 187-ФЗ (целостность исходного кода) | Прослеживаемость влияния ИИ на исходный код | `git_commit_before/after` на каждый LLM-вызов даёт полную трассу изменений репозитория | `SELECT git_commit_before, git_commit_after FROM pipeline_audit WHERE run_id=$1` | ТР |

## 2. Сертификация ФСТЭК

Формальная сертификация средства по требованиям ФСТЭК России и регистрация
объекта в реестре значимых объектов КИИ выделены в отдельную задачу плана
улучшений — **R15**. Система технологически готова: документация соответствует
III категории значимости, технические механизмы защиты реализованы и
проверяемы (см. таблицу выше). Сама процедура сертификации и регистрации
выполняется субъектом КИИ.

## 3. Контрольные команды (audit drill, < 1 мин)

```bash
# Существование таблицы
docker compose exec postgres psql -U postgres -d contextstore \
  -c "\dt pipeline_audit"

# Append-only — должно отказать
docker compose exec -e PGPASSWORD=multiagent_audit_pwd postgres \
  psql -U multiagent_app -d contextstore \
  -c "DELETE FROM pipeline_audit"
# → ERROR:  permission denied for table pipeline_audit

# Запись доступна
docker compose exec -e PGPASSWORD=multiagent_audit_pwd postgres \
  psql -U multiagent_app -d contextstore \
  -c "SELECT count(*) FROM pipeline_audit"

# Воспроизведение одного прогона
docker compose exec multiagent python -m app.cli.multiagent_replay \
  --run-id=<UUID>

# Манифест без LLM-вызова (для аудита запросов)
docker compose exec multiagent python -m app.cli.multiagent_replay \
  --run-id=<UUID> --no-call --json

# Air-gapped-проверка (для объектов КИИ высших категорий)
docker compose exec multiagent curl --max-time 5 https://api.anthropic.com
# → connection refused / network unreachable
```

## 4. Что не покрыто текущей реализацией (отложенные работы)

1. **Анонимизация персональных данных** в `full_prompt`/`full_completion` — в требование пользователя потенциально могут попасть ПДн. В on-prem-контуре допустимо; для публичного SaaS требуется отдельный модуль редактирования запросов.
2. **Шифрование `full_prompt`/`full_completion` at rest** — сейчас данные хранятся в PostgreSQL в открытом виде. Решение: прозрачное шифрование tablespace или шифрование на уровне приложения.
3. **Подписанные коммиты** — `git_commit_before/after` фиксируется, но коммиты не подписаны GPG.
4. **Контроль изменений конвенций проекта между прогонами** — при изменении `project_conventions.json` воспроизведение может дать расхождение; решение — фиксировать SHA-256 от bundle в метаданных.
5. **Автоматический алертинг на расхождение при воспроизведении** — сейчас проверка ручная (CLI); в production целесообразен ночной job с уведомлением.
6. **Интеграция с ГосСОПКА** для субъектов КИИ — внешний интерфейс на стороне заказчика, не часть системы.
7. **ГОСТ-криптография** (ГОСТ Р 34.11-2012 «Стрибог») для объектов высших категорий — сейчас используется SHA-256; адаптация под конкретного клиента возможна.
