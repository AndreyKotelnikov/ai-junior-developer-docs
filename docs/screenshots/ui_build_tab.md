# Вкладка «Сборка»

> Изображение экрана прилагается отдельным файлом. Ниже — текстовое описание и анализ.

## Что показано

Вкладка **«Сборка»** в центральной рабочей области отображает результат
компиляции доработанного проекта командой `dotnet build`: статус сборки
(`success` / `failed`) и тренд числа ошибок компилятора (CS-ошибок) по итерациям
fix-loop.

Под областью сборки раскрывается блок **`TaskExplanation`** — пост-объяснение от
агента `ExplainerAgent`. Это структурированный JSON-ответ в формате «Что сделано
/ Зачем / Что проверить» (объём ограничен ≈ 400 токенами ответа). Пример полей
ответа:

```json
{
  "what": "Добавлена сущность CustomerCategory с 4 полями и self-referencing FK.",
  "why": "Требование декомпозировано Orchestrator на подзадачи; выполнено Coder.",
  "verify": "Проверить миграцию EF Core и регистрацию ассоциации в Initializer."
}
```

## Анализ

Вкладка связывает технический результат (успешная сборка) с человекочитаемым
объяснением. `TaskExplanation` формируется отдельным агентом `ExplainerAgent`
уже после успешной сборки и записывается, в том числе, в поле `subtask_why`
журнала `pipeline_audit` — то есть объяснение «зачем» становится частью
проверяемого аудиторского следа, а не остаётся только в интерфейсе. Тренд
CS-ошибок наглядно показывает работу цикла исправлений: число ошибок снижается
от итерации к итерации до нуля.
