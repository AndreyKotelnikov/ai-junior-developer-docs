# Сценарий А — Бизнес-пользователь: полное описание

**Назначение**: пошагово описать end-to-end сценарий работы бизнес-пользователя (разработчик .NET в КИИ-операторе, тимлид, бизнес-аналитик) — от ввода требования на естественном языке до утверждённого diff с пост-объяснением.
**Связанные документы**:
- [`../architecture/ui_README.md`](../architecture/ui_README.md) — описание веб-интерфейса.
- [`scenario_b_compliance_officer_full.md`](scenario_b_compliance_officer_full.md) — сценарий compliance-офицера.
- [`scenario_c_onboarding_admin_full.md`](scenario_c_onboarding_admin_full.md) — сценарий администратора онбординга.

> Сценарий А — основной из трёх непересекающихся пользовательских сценариев MVP. Точка входа — страница `Agents`; узловая точка контроля человека — `ApprovalGate`; артефакт результата — diff + `TaskExplanation` + `run_id`. Целевая длительность — ≈ 5 минут.

---

## 1. Роль и предусловия

| Параметр | Значение |
|---|---|
| Роль | Бизнес-пользователь: разработчик .NET, тимлид, бизнес-аналитик КИИ-оператора (банк, оборона, госсектор) |
| Точка входа | страница `Agents` веб-интерфейса (`http://localhost:8501/`) |
| Предусловия | целевой проект `dotnet-app` подключён; конвенции (`project_conventions.json`) активированы (см. Сценарий В) |
| Узловая точка | `ApprovalGate` — утверждение плана человеком (human-in-the-loop) |
| Артефакт результата | unified diff + `TaskExplanation` + `run_id` прогона |
| Целевая длительность | ≈ 5 минут |

---

## 2. Шаги сценария

### Шаг 1. Открыть страницу `Agents`

Пользователь открывает веб-интерфейс на порту 8501. Главная страница `Agents` служит визуальным якорем: на одном экране — поле ввода требования, шкала четырёх ролей агентов (`Explorer / Orchestrator / Coder / Tester`), сводная панель метрик и трёхколоночная рабочая область (`SubtaskList` / `CenterTabs` / `MetricsPanel`).

### Шаг 2. Ввести бизнес-требование

В поле `RequirementInput` пользователь формулирует требование на естественном русском языке (длиной не более 200 слов). Пример:

> «Добавь новую сущность `CustomerCategory` с 4 полями: Название (string), Код (string, уникальный), Приоритет (int), РодительскаяКатегория (self-referencing FK).»

RAG-классификатор (Zero-Retrain Router) присваивает требованию бейдж типа задачи (`add_entity`). Опционально можно выбрать шаблон требования из 8 категорий.

### Шаг 3. Выбрать режим запуска

В панели настроек выбирается режим LLM (облачный `API` / локальный `Ollama`), число RAG-примеров (0 / 3 / 5) и режим запуска (`Real` — реальная генерация, `Demo` — воспроизведение записанного прогона).

### Шаг 4. Запустить пайплайн

Нажатие `[Запустить]` отправляет `POST /api/tasks`. Запускается восьмифазный пайплайн (LangGraph-DAG): Phase 0 (RAG-классификация) → Phase 1 (планирование Orchestrator) → Phase 2 (Roslyn-локализация, 0 LLM) → Phase 3 (кодирование Coder) → Phase 3.5 (Roslyn-валидация) → Fix-loop → Phase 4 (сборка `dotnet build`) → Phase 5 (пост-объяснение). Прогресс отображается в реальном времени через WebSocket-канал `/ws/progress/{task_id}`.

### Шаг 5. Просмотр результата

На вкладке `Код` (`CenterTabs`) пользователь видит diff по файлам с hunk-based вьюером; вкладка `Сборка` показывает статус `dotnet build` и тренд CS-ошибок; вкладка `План` — DAG подзадач.

### Шаг 6. Утвердить через `ApprovalGate` и получить `TaskExplanation`

Если `approvalRequired = true`, перед кодированием план требует утверждения человеком через баннер `ApprovalGate` на вкладке `План`. После завершения под рабочей областью раскрывается блок `TaskExplanation` от `ExplainerAgent` в формате «Что сделано / Зачем / Что проверить» (≤ 400 completion-токенов). Пользователь получает unified diff, `TaskExplanation` и `run_id` прогона.

---

## 3. Артефакты и журналирование

| Артефакт | Где доступен |
|---|---|
| Unified diff | вкладка `Код`; экспорт `[.patch]` |
| `TaskExplanation` | блок под рабочей областью; поле `pipeline_audit.subtask_why` |
| `run_id` | идентификатор прогона; ключ для Сценария Б (аудит) |
| Markdown-отчёт | экспорт `[.md]` |

На каждом LLM-вызове параллельно срабатывает sidecar-журналирование в `pipeline_audit` (append-only). Итоговая запись доступна compliance-офицеру по `run_id` (см. Сценарий Б).

---

## 4. Контроль человека (human-in-the-loop)

Узловая точка `ApprovalGate` обеспечивает требование контроля человека над ИИ (Указ Президента РФ № 490). План декомпозиции от `Orchestrator` с полем `why` по каждой подзадаче предъявляется пользователю до начала кодирования; кодирование стартует только после явного утверждения.

---

## 5. Известные ограничения

- Длина требования ограничена 200 словами; более длинные требования рекомендуется декомпозировать.
- Режим `Demo` воспроизводит ранее записанный прогон и не выполняет реальных LLM-вызовов.
- Для задач уровня «сложная» (5–7 файлов, новая сущность с FK) целевое `TimeToSuccess` — ≈ 20 минут, что превышает базовую оценку сценария в 5 минут.
