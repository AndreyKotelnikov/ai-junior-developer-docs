# Полное описание конвенций специализированного корпоративного фреймворка

## Назначение

Полный каталог неявных конвенций оформления кода специализированного корпоративного фреймворка, используемого в целевом приложении `dotnet-app` (ASP.NET Core + EF Core Code-First). Документ раскрывает пласт правил, не отражённых в публичной документации C# / EF Core и обнаруживаемых только через анализ существующих регистраций.

## Связанные документы

- `docs/architecture/PROJECT_ARCHITECTURE.md` — структурный паспорт проекта.

## 1. Контекст

Корпоративные приложения уровня enterprise добавляют поверх стандартных конвенций C# и EF Core дополнительный пласт неявных правил. Нарушение этих правил не всегда даёт ошибку компиляции — часть из них приводит к скрытым дефектам конфигурации модели данных, обнаруживаемым только во время выполнения. Автоматическая доработка такого кода требует кодирования конвенций как параметризованных шаблонов в составе AST-трансформеров.

## 2. Каталог конвенций

### 2.1. Именование полей внешнего ключа

- Поле внешнего ключа именуется с обязательным суффиксом `ID` в верхнем регистре: `CustomerID`, `OrderID` (а не `CustomerId` / `customer_id`).
- Тип поля FK — соответствующий тип первичного ключа целевой сущности.
- Навигационное свойство именуется по имени целевой сущности без суффикса: `public Customer Customer { get; set; }`.

### 2.2. Регистрация ассоциаций

- Ассоциации между сущностями регистрируются **на стороне родительской (целевой) сущности**, а не дочерней.
- Связь «один-ко-многим» регистрируется как `AddOneToManyAssociation<ChildEntity>(p => p.NavProperty)` в регистрации родительской сущности.
- Связь «один-к-одному» для расчётных под-сущностей регистрируется как `AddOneToOneAssociation<T>()` — без лямбда-выражения.
- Для lookup-полей (ссылка на справочную сущность) ассоциация регистрируется на стороне сущности-справочника (TARGET), а не на стороне ссылающейся сущности.

### 2.3. Централизация конфигурации

- Запрещено создавать отдельные конфигурационные файлы вида `{Entity}Configuration.cs`: вся регистрация свойств, индексов и ассоциаций централизована в едином файле `DataConfiguration.cs` (около 907 строк, ~150 регистраций ассоциаций для целевого приложения).
- Запрещено вручную добавлять поля `DbSet<T>` в класс контекста базы данных — регистрацию `DbSet` выполняет базовый фреймворк автоматически на основе записей `Initializer.cs`.

### 2.4. Регистрационный конвейер `Initializer.cs`

- Каждая сущность регистрируется вызовом `AddEntity<T>()` с последующей цепочкой методов конфигурации (`AddDefaultMnemonic()` и др.).
- Порядок регистраций в `Initializer.cs` значим: родительские сущности регистрируются раньше зависимых.

### 2.5. Прочие требования

- BOM-маркер UTF-8 может встречаться в `.cs`-файлах в неожиданных позициях; инструменты доработки кода обязаны корректно обрабатывать BOM при чтении и поиске.
- Энумерации и примитивные скалярные свойства не требуют записей в `DataConfiguration.cs` / `Initializer.cs`; регистрации требуют только навигационные свойства и ассоциации.

## 3. Каскад правок

Из перечисленных конвенций следует, что добавление навигационного свойства в сущность затрагивает каскад из 4 ключевых артефактов модели данных (`Entity`, `DbContext`, `Initializer.cs`, `DataConfiguration.cs`) и расширяется до 5–7 файлов с учётом связанных CRUD-сервисов и REST-контроллеров.
