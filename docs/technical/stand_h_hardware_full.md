# Стенд H: аппаратная конфигурация и характеристики air-gapped-инференса

Стенд H — референсная air-gapped (физически изолированная) конфигурация
факторного эксперимента: мультиагентная система целиком работает на локальной
языковой модели без обращения к внешним API. Именно на этом стенде проверяется
гипотеза о паритете локального разворачивания модели с облачной.

## 1. Аппаратная конфигурация

| Параметр | Значение |
|---|---|
| GPU | NVIDIA GeForce RTX 3080 Ti Laptop |
| Видеопамять (VRAM) | 16 ГБ |
| Архитектура / Compute Capability | Ampere, 8.6 |
| Пропускная способность памяти | 512 ГБ/с |
| Роль конфигурации | нижняя граница реалистичного локального сценария |

Выбор минимально достаточной конфигурации обоснован: если на 16 ГБ VRAM
достигается целевой паритет, то на более мощном железе результат не ухудшается по
допущению монотонности VRAM-bound инференса. Опциональная замена на 32B-модель на
GPU ≥ 24 ГБ зафиксирована как задача R9.

## 2. Локальная модель и параметры инференса

| Параметр | Значение |
|---|---|
| Модель (все роли стенда H) | `qwen2.5-coder:14b` |
| Квантизация | Q4_K_M |
| Потребление VRAM | около 9 ГБ |
| Лицензия | Apache 2.0 |
| Движок | Ollama (air-gapped, без внешних вызовов) |
| Окно контекста (`OLLAMA_NUM_CTX`) | 6144 токена (из стандартных 12 K) |
| `temperature` | 0 |

Ограничение контекста до 6 K токенов обосновано эффектом потери середины
контекста (Liu et al., 2024): при бюджете 6 K локальная модель не страдает от
позиционной деградации внимания.

В факторном эксперименте все локальные роли стенда H используют единую модель
`qwen2.5-coder:14b`; вывод о паритете относится именно к ней. В production
air-gapped-конфигурации применяется per-agent-маршрутизация (Orchestrator —
`qwen3:14b`; Coder, FixAgent, RoslynExplorer, Tester — `qwen2.5-coder:14b`;
ExplainerAgent — `qwen2.5:3b`); её формальная валидация — задача R9.

## 3. Сериализация вызовов на одной видеокарте

Одновременные параллельные LLM-вызовы из нескольких агентов на одной видеокарте
16 ГБ приводили к исчерпанию памяти (OOM) и аварийному завершению контейнера
Ollama. Поэтому все вызовы сериализуются через очередь с одним рабочим потоком и
приоритизацией по ролям.

Параметры движка:

| Параметр Ollama | Значение |
|---|---|
| `OLLAMA_NUM_PARALLEL` | 1 |
| `OLLAMA_MAX_LOADED_MODELS` | 1 |
| `OLLAMA_KEEP_ALIVE` | 15 минут |

Приоритеты агентов в очереди (меньшее значение — выше приоритет):

| Агент | Приоритет |
|---|:-:|
| Orchestrator | 1 |
| RoslynExplorer | 2 |
| Coder | 3 |
| FixAgent | 5 |
| Tester | 5 |
| ExplainerAgent | 7 |

Приоритизация даёт первоочередную обработку планирования и локализации
(критический путь), что сокращает простой остальных ролей.

## 4. Характеристики производительности

| Показатель | Значение |
|---|---|
| Эталонное время LLM-части простой однофайловой задачи | ≈ 50,5 с (целевой порог ≤ 150 с) |
| Аварийные завершения по OOM при параллельных вызовах | снижены с ≈ 18 % до 0 % после сериализации |
| Событий загрузки модели в VRAM на задачу | 1 (модель удерживается в памяти `KEEP_ALIVE`) |
| Воспроизводимость (σ Pass@1 по 5 повторам стенда H) | 5,58 п.п. — на уровне облачных стендов |

Эталонное время LLM-части не следует путать с метрикой полного успешного прогона
(она включает все итерации цикла исправлений).

## 5. Воспроизведение air-gapped-инстанса

1. Загрузить модель в локальный реестр Ollama: `ollama pull qwen2.5-coder:14b`.
2. Задать параметры окружения: `OLLAMA_NUM_CTX=6144`, `OLLAMA_NUM_PARALLEL=1`,
   `OLLAMA_MAX_LOADED_MODELS=1`, `OLLAMA_KEEP_ALIVE=15m`.
3. Развернуть стенд H: `docker compose -f docs/experiment/stand_configs/docker-compose.stand-h.yml up`.
4. Запустить прогон: `python docs/experiment/hyp-run-tests.py --stand H`.

Контур не требует доступа в интернет: модель и все сервисы работают локально.
