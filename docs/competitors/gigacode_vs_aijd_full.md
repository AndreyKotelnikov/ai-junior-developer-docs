## 1. Описание конкурента

В числе российских on-premise-решений в области AI-кодогенерации главным конкурентом является **GigaCode Enterprise on-premise** производства СберТеха. Одна из последних версий — GigaCode 3.0 (ноябрь 2025 г.) с моделями GigaCode Inline 4.0 13B-A3B (MoE-архитектура для FIM-задач автодополнения) и CodeChat на дообученной open-source-модели 235B с контекстным окном 131 K токенов; поддерживает 35+ языков программирования, агентный режим (Agent Mode) и субагенты в составе SWE-агентов (с ноября 2025 г.); ClickScore Python 86,4 %, LiveCodeBench 65 %, SWE-Bench 21,33 %.

## 2. Принципиальное различие классов продуктов

**GigaCode Enterprise и AI Junior Developer — продукты принципиально разных классов.** GigaCode позиционируется как массовый AI-ассистент разработчика в формате IDE-co-pilot уровня GitHub Copilot / Cursor — единый инструмент для повседневной работы тысяч разработчиков (автодополнение, чат, объяснение, ревью, агентный режим внутри сессии); AI Junior Developer — узкоспециализированная мультиагентная нейросимвольная фабрика автоматической end-to-end-доработки enterprise-приложений на стеке ASP.NET Core 3.1 + EF Core, где LLM выступает лишь одним из 7 агентов, а на эталонной инструментальной задаче (доработка существующей сущности) приблизительно 92 % подзадач (12 из 13) решено детерминированными Roslyn-AST-трансформерами без обращения к модели. Замеренное покрытие по другим типам задач пока не получено и является направлением дальнейшей работы. Это **не конкуренты «лоб в лоб», а взаимодополняющие классы инструментов**: GigaCode — для повседневной работы разработчика в IDE, AI Junior Developer — для регламентированной автогенерации в зоне строгого compliance.

## 3. Сильные стороны GigaCode

**Сильные стороны GigaCode объективно превосходят AI Junior Developer по масштабу, экосистеме и широте.** Сбер обучает Inline-модель с нуля на 5 трлн токенов; присутствует полная экосистема (GitVerse + GigaIDE + GigaCode + GigaStudio + GigaView); по данным А. Белевцева на GigaConf 2025 — более 45 тыс. установок плагина и около 25 тыс. активных пользователей в месяц; интеграция с семейством JetBrains, Visual Studio Code, Android Studio, Jupyter; 35+ языков против одного стека (.NET) у настоящей системы. Сайт продукта декларирует «Ускорение написания кода на 25 %» (platformv.sbertech.ru). На массовых задачах автодополнения GigaCode объективно сильнее любого решения, использующего Qwen2.5-Coder:14b в качестве базовой модели.

## 4. Восемь решающих критериев для регулируемых сегментов

**Уникальные отличия AI Junior Developer от GigaCode сводятся к восьми пунктам, имеющим решающее значение для регулируемых сегментов** (банки, КИИ, госорганы).

### 4.1. Вертикальная интеграция в задачи внедрения CRM/ERP-систем

В отличие от горизонтального позиционирования GigaCode (универсальный co-pilot для тысяч разработчиков и 35+ языков), AI Junior Developer спроектирован как вертикально интегрированный конвейер для типовых, относительно простых и высоко детерминированных задач при внедрении CRM/ERP-решений на ASP.NET Core + EF Core: добавление сущности с навигационными свойствами, регистрация ассоциаций, добавление CRUD-сервиса по шаблону, добавление REST-endpoint, валидация поля, миграция БД. Эти задачи характеризуются плотной семантической зависимостью между 5–7 файлами и наличием неявных проектных конвенций — именно тот класс, где детерминированные AST-трансформеры дают наибольший ROI и где универсальный IDE-co-pilot объективно избыточен.

### 4.2. Детерминированность подзадач без LLM

Главное архитектурное преимущество — нейросимвольный подход. На эталонной инструментальной задаче «доработка существующей сущности» (12 подзадач из 13 решены без LLM, что соответствует приблизительно 92 % покрытия) только малая часть подзадач решается через LLM, остальное — через 10 программных Roslyn-AST-трансформеров и 5 фиксеров типовых CS-ошибок. **Важно:** замер «92 % без LLM» получен на одной эталонной задаче типа «модификация сущности»; по другим типам задач (модификация сервиса, фильтрация, импорт / экспорт и др.) подробных замеров не проводилось — повышение этого показателя по разнообразным классам задач является направлением дальнейшей работы. У GigaCode каждое автодополнение и каждая итерация агента — это отдельный LLM-вызов, что (а) даёт меньшую воспроизводимость результатов; (б) даёт большую вариативность ответа на одинаковом входе; (в) повышает стоимость каждой операции; (г) усложняет аудит решений системы.

### 4.3. Compliance-grade audit log и побайтовая воспроизводимость

AI Junior Developer реализует то, чего у GigaCode публично нет: append-only PostgreSQL-журнал с REVOKE на UPDATE/DELETE/TRUNCATE на уровне СУБД; CLI-инструмент `multiagent_replay` для повторного выполнения сохранённого прогона; опциональная нотаризация SHA-хэшей в OpenTimestamps как криптографическое доказательство времени записи. **Важно** относительно показателя побайтовой воспроизводимости: целевая величина для Ollama-сценария — 100 % byte-identical replay при фиксации параметров инференса (`temperature = 0`, `seed`); это **плановое (теоретически достижимое) значение**, к которому проект пока стремится. Фактически в текущей реализации параметр `seed` явно не фиксируется, и эксперименты на Ollama дали разброс результатов между повторами; на стороне Anthropic фактический показатель составил приблизительно 50 % (что соответствует ограничению `temperature` облачного провайдера). Для банка/госоргана архитектурный механизм append-only-журнала + replay уже закрывает требования Приказа ФСТЭК № 239 (защита значимых объектов КИИ) и требований Банка России к аудиту автоматизированных систем; повышение показателя побайтовой идентичности до 100 % — направление дальнейшей работы (задача R10: явная фиксация `seed` в `OllamaScheduler`).

### 4.4. Существенно низкие аппаратные требования on-premise

Single-GPU 16 ГБ (RTX 3080 Ti Laptop в эксперименте; рекомендуемый 24 ГБ — RTX 4090) против ориентировочного кластера из 4–8 серверных GPU класса A100/H100 80 ГБ для GigaCode. В рублёвом эквиваленте — порядка 250 тыс. ₽ против десятков миллионов. Это делает AI Junior Developer доступным для пилотных проектов в air-gapped-контурах без выделенного AI-кластера и для регулируемых организаций, не готовых вкладывать в GPU-инфраструктуру до обоснования ROI.

### 4.5. Сверка с российской регуляторной базой (предварительная, без сертификации)

AI Junior Developer **поверхностно сверен** с основными публичными требованиями российского регулирования: 152-ФЗ (защита ПДн), 187-ФЗ (КИИ), Приказы ФСТЭК № 235 (АС ГИС) и № 239 (защита КИИ), ГОСТ Р 34.11-2012 «Стрибог» (для целостности логов), Принципы доверенного ИИ Альянса в сфере ИИ. Архитектурные механизмы проекта (append-only Postgres-журнал с REVOKE на DML, SHA-256-хеши, OpenTimestamps, replay CLI, ApprovalGate, ExplainerAgent) **спроектированы с учётом** этих требований. **Важно:** детальной проверки на соответствие всем требованиям перечисленных нормативных актов в полном объёме пока не проводилось; **сертификатов и формальных подтверждений от соответствующих регуляторов (ФСТЭК России, ФСБ России, Минцифры и др.) у проекта на текущий момент нет**. Получение сертификата ФСТЭК России на уровне доверия 4 или 5 запланировано как задача R15 после защиты (оценочный срок не менее 6 месяцев). У GigaCode декларируется только запись в Реестре российского ПО (для целей ФЗ-44/223) и общие фразы про «безопасность on-prem»; сертификат ФСТЭК России в государственном реестре сертифицированных СЗИ для GigaCode также не найден; Astra/РЕД/Альт-совместимость публично не подтверждена.

### 4.6. Проектно-агностичная архитектура и быстрый онбординг

Документ `project_conventions.json` + конвейер auto-discovery (AST-mining + git-cochange-анализ) дают онбординг нового .NET-проекта приблизительно за 4 часа против ручной настройки 2–3 дня — приблизительно восемнадцатикратное ускорение. У GigaCode подключение проекта происходит через построение RAG-индекса, но конкретные сроки и метрики онбординга публично не раскрываются.

### 4.7. Объяснимость (XAI) встроена в pipeline

ExplainerAgent генерирует post-hoc объяснение «что сделано / зачем / что проверить» для каждой задачи в формате машиночитаемого аудит-артефакта с фиксацией в `pipeline_audit.subtask_why`. Это обязательное требование Принципов доверенного ИИ и Приказа ФСТЭК № 117. У GigaCode объяснение кода (`/explain`) есть как функция чата для разработчика, но это объяснение «человеком», а не структурированный аудит-артефакт.

### 4.8. Предсказуемая стоимость одной задачи

Приблизительно 0,10 долл. США за задачу на Claude Haiku 4.5 (после оптимизации Slim Coder с prompt caching: 7 K → 2,5 K токенов) или 0 долл. на Ollama. У GigaCode стоимость на одну операцию публично не раскрывается; SaaS-тариф «от 1 699 ₽/мес/пользователя» (по aimarketcap.ru, aineuro-tools.ru) измеряется в подписной форме, а не на единицу полезной работы; on-premise — по подписной модели за конкурентную лицензию (точная цена за единицу не публикуется, минимальная закупка 10 лицензий на 1 год).

## 6. Стратегическое позиционирование

Для крупного банка / госкорпорации обоснованный практический сценарий — **сосуществование** обоих продуктов: GigaCode Enterprise (on-premise) как корпоративный AI-ассистент уровня IDE для всех разработчиков (повседневная работа, автодополнение, чат, объяснение); AI Junior Developer — отдельный регламентированный конвейер для определённого класса задач (типовая доработка legacy ASP.NET Core микросервисов: добавление endpoint, валидация поля, миграция БД), где требуется автоматизация end-to-end с audit-grade-журналом. Это не «или–или», а «и»: один и тот же разработчик пишет общий код в IDE с GigaCode, а на регламентированном CRUD-классе задач запускает AI Junior Developer.
