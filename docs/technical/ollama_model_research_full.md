# Полное исследование выбора локальной LLM-модели для on-prem-конфигурации

## Назначение

Расширенная версия исследования выбора локальной LLM-модели для on-premise-конфигурации (air-gapped-контур). Документ раскрывает критерии отбора и сравнительный анализ 8 кандидатов open-source-моделей кодогенерации.

## Связанные документы

- `docs/technical/stand_h_hardware_full.md` — аппаратная конфигурация стенда локального инференса.

## 1. Постановка задачи

Гипотеза о локальной развёртке (H4) требует выбора open-source-модели кодогенерации для air-gapped-инстанса без внешних API-вызовов. Целевое ограничение — нижняя граница реалистичного локального сценария: одна GPU с 16 ГБ видеопамяти (NVIDIA RTX 3080 Ti Laptop, Ampere 8.6, 512 ГБ/с).

## 2. Критерии отбора

| Критерий | Требование |
|---|---|
| Лицензия | Допускающая коммерческое использование в РФ (предпочтительно Apache 2.0) |
| Размещение в VRAM | Модель в выбранной квантизации помещается в 16 ГБ с запасом на KV-кэш |
| Специализация | Обучение преимущественно на коде; поддержка C# |
| Контекстное окно | Достаточно для anchor + signatures + сжатого графа (бюджет 6 K токенов на Ollama) |
| Качество кодогенерации | Конкурентные результаты на code-бенчмарках для своего класса параметров |

## 3. Кандидаты (8 моделей)

Рассматривались 8 кандидатов open-source-моделей кодогенерации. Для каждого проверялось соответствие пяти критериям отбора и фактическое потребление видеопамяти в квантизации Q4_K_M.

| Модель | Семейство | Лицензия | VRAM в Q4_K_M | Примечание по качеству |
|---|---|---|---|---|
| `qwen2.5-coder:14b` | Qwen2.5-Coder | Apache 2.0 | ~9 ГБ | Выбран — баланс качества и VRAM |
| `qwen2.5-coder:32b` | Qwen2.5-Coder | Apache 2.0 | ~20 ГБ | Не помещается с запасом в 16 ГБ |
| `qwen2.5-coder:7b` | Qwen2.5-Coder | Apache 2.0 | ~5 ГБ | Ниже по качеству |
| `codellama:13b` | Code Llama | Llama Community | ~8 ГБ | Слабее на C# |
| `deepseek-coder:6.7b` | DeepSeek-Coder | DeepSeek License | ~5 ГБ | Ограничения лицензии |
| `deepseek-coder:33b` | DeepSeek-Coder | DeepSeek License | ~20 ГБ | Не помещается |
| `starcoder2:15b` | StarCoder2 | BigCode OpenRAIL-M | ~9 ГБ | Лицензия с ограничениями использования |
| `granite-code:8b` | Granite Code | Apache 2.0 | ~5 ГБ | Слабее на профильных задачах |

Победитель отбора — `qwen2.5-coder:14b` в квантизации Q4_K_M (~9 ГБ) на 16 ГБ VRAM.

## 4. Результат отбора

Оптимальным кандидатом для нижней границы локального стека выбран **`qwen2.5-coder:14b` в квантизации Q4_K_M**:

- **Лицензия** — Apache 2.0, без ограничений на коммерческое применение в РФ.
- **Размещение в VRAM** — около 9 ГБ потребления видеопамяти, что оставляет запас в пределах 16 ГБ на KV-кэш и контекст.
- **Специализация** — семейство Qwen2.5-Coder обучено на корпусе 5,5 трлн токенов; модель 32B-Instruct классифицируется в обзорах как SOTA среди open-source (73,7 % Aider), 14B-вариант выбран как компромисс между качеством и потреблением видеопамяти (Qwen2.5-Coder Tech Report, Hui B. et al., arXiv:2409.12186, 2024).
- **Контекстное окно** — достаточно для методологического бюджета 6 K токенов на Ollama (`OLLAMA_NUM_CTX = 6144`), обоснованного эффектом Lost in the Middle.

## 5. Принцип монотонности

Конфигурация 16 ГБ — методологически наиболее жёсткий стенд (нижняя граница). Если на 16 ГБ гипотеза H4 о паритете подтверждается, на бо́льшем железе результат не ухудшается по принципу монотонности VRAM-bound инференса. Опциональная замена 14B → 32B на железе ≥ 24 ГБ зафиксирована как задача R9.
